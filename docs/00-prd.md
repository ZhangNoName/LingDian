## 2. 项目背景

餐饮商家通常同时面对堂食点餐、扫码点餐、外带自提、门店活动、会员营销和订单履约等场景。传统做法常见问题：

- 前端多端割裂，主题、配置和交互不统一
- 点餐、门店、订单、营销能力分散，维护成本高
- 业务变化频繁，但系统扩展性不足
- 后续新增 App 时需要重复建设

因此需要建设一个点餐一体化平台，以统一的后端服务承接多端业务，并建立统一品牌主题与配置体系。

## 3. 产品目标

### 3.1 业务目标

- 提供标准化点餐闭环：浏览菜单、下单、支付、履约、复购
- 支持门店业务配置统一管理
- 为后续会员、营销、经营分析能力预留扩展空间

### 3.2 系统目标

- Web 与 uni-app 共享统一颜色配置和品牌主题
- 后端统一承载核心业务域，避免多套服务分裂
- 构建可扩展的多端架构，后续新增 App 复用现有业务层

### 3.3 首期交付目标

- 完成项目技术框架初始化
- 明确首期 MVP 范围
- 建立核心业务模块边界
- 输出可进入研发实施的 PRD

## 4. 用户角色

### 4.1 C 端用户

- 普通顾客：扫码/进入小程序进行点餐、支付、查看订单
- 会员用户：使用优惠券、积分、会员价、充值卡等权益

### 4.2 B 端用户

- 门店店长：配置门店营业状态、菜单、活动内容
- 店员/收银员：处理订单、核销、自提叫号、售后
- 运营人员：查看门店数据、管理活动、配置营销

### 4.3 平台管理员

- 管理品牌级配置、权限、门店、商品基础档案

## 5. 产品范围

### 5.1 首期 MVP 范围

- 门店识别与门店状态展示
- 菜单分类与商品列表
- 商品详情、规格、加料、备注
- 购物车
- 下单与订单确认
- 订单状态查询
- 我的页面基础能力
- 门店配置与菜单配置的接口骨架

### 5.2 二期扩展范围

- 优惠券、满减、折扣活动
- 会员体系、积分体系
- 自提叫号
- 小票打印、厨房打印
- 配送对接

- 多门店与品牌总部视角

### 5.3 暂不纳入首期

- 复杂加盟结算
- ERP 深度集成
- 供应链采购系统
- 多国家多语言多币种

## 6. 业务流程

### 6.1 C 端点餐主流程

1. 用户进入小程序或 H5
2. 系统识别门店、桌台码、渠道来源
3. 加载门店营业状态与菜单
4. 用户选择商品、规格、数量并加入购物车
5. 用户确认订单信息
6. 进行支付或提交待支付订单
7. 后端生成订单并流转履约状态
8. 用户在订单页查看状态、取餐信息、售后入口

### 6.2 门店运营流程

1. 店长登录 Web 运营端
2. 配置营业状态、菜单上架、活动内容
3. 查看订单流转情况
4. 按需调整库存、售罄、出餐状态

- `completed`

### 7.5 用户中心模块

功能说明：

### 8.1 uni-app 用户端

定位：

- 作为 C 端核心点餐入口
- 优先支持微信小程序与 H5
- 后续平滑扩展为 App

核心页面：

- 首页
- 点餐页
- 商品详情弹层
- 购物车
- 订单确认页
- 订单列表/详情

### 8.2 Web 端

定位：

- 品牌门户、活动展示、门店展示
- 后续可演进为运营管理端或管理后台壳

核心页面：

- 项目首页/门户首页
- 门店与活动展示页
- 运营配置入口

## 10. 设计与品牌规范

### 10.1 主题原则

- `theme/` 目录为颜色单一来源
- Web 使用 `theme/colors.css`
- uni-app 使用 `theme/colors.scss`
- 所有主题调整先改 `theme/`，不允许多端各自维护主色

### 10.2 当前主题方向

- 主色：暖橙色，突出食欲感与转化感
- 辅色：橙红色系，用于强调活动和转化按钮
- 背景：暖白色，提升餐饮场景亲和感

### 10.3 设计约束

- 关键 CTA 必须使用统一主色
- 订单状态颜色需标准化
- Web 与 uni-app 的品牌识别必须保持一致

## 11. 后端架构要求

### 11.1 技术选型

- Node.js
- NestJS
- REST API 为首期主接口风格
- 后续可按需要扩展 WebSocket、消息队列、任务调度

### 11.2 模块划分建议

- `auth`：登录授权
- `users`：用户中心
- `stores`：门店模块
- `menu`：菜单模块
- `cart`：购物车模块
- `orders`：订单模块
- `payments`：支付模块
- `promotions`：营销模块
- `memberships`：会员模块

### 11.3 非功能要求

- 接口统一前缀与版本管理
- 关键接口支持参数校验
- 支持跨端调用
- 支持环境变量配置
- 便于后续接入 Swagger

## 13. 接口域清单

首期建议接口域：

- `/stores/current`：当前门店信息
- `/menu/current`：当前门店菜单
- `/cart`：购物车读写
- `/orders`：创建订单、订单详情、订单列表
- `/users/profile`：用户信息
- `/marketing/banners`：活动 banner

## 14. 权限与安全

### 14.1 C 端

- 基于用户身份访问订单、优惠券、个人资料

### 14.2 B 端

- 按角色区分店长、店员、运营、管理员

### 14.3 安全要求

- 核心接口鉴权
- 订单金额服务端校验
- 活动优惠不能由前端直接决定最终结果
- 门店隔离与数据权限控制

## 15. 埋点与数据分析

首期关键埋点：

- 首页曝光
- 商品点击
- 加入购物车
- 提交订单
- 支付成功
- 再来一单

核心指标：

- UV / 下单人数
- 加购率
- 下单转化率
- 支付转化率
- 客单价
- 复购率

## 16. 里程碑建议

### M1 项目初始化

- 完成仓库结构
- 完成主题统一
- 完成基础页面骨架
- 完成后端核心模块初始化

### M2 MVP 开发

- 打通门店、菜单、购物车、订单主链路
- 打通登录与订单查询

### M3 运营增强

- 上线优惠券、活动推荐位、经营配置

### M4 多端扩展

- App 端适配
- 多门店品牌化能力增强

## 17. 风险与注意事项

- 若 Web 和 uni-app 各自维护主题，后期品牌一致性会快速失控
- 若后端先按单体需求硬编码，后续会员和营销扩展成本会变高
- 若订单金额计算放在前端，存在较高风险
- 若首期一次性做过多运营能力，MVP 周期会被拉长

## 18. 当前结论

本项目建议以“统一主题 + uni-app 用户端优先 + NestJS 后端统一承载 + Web 作为运营与展示扩展端”的方式推进。  
首期聚焦点餐主链路，先保证可下单、可查看订单、可配置门店，再逐步扩展营销和会员能力。
